# NavalCommand v0.2.0 Release Plan

Based on the architectural directives set forth in the initial specification (`AGENTS.md`) and the completion of the core v0.1.0 foundational release, the following priority backlog items have been scoped for the `v0.2.0` minor release.

## 1. Document Exporting & Report Generation (`feature/v0.2-document-export`)
* **Objective:** Enable Officers to generate standard military reports (Personnel Rosters, Attendance Summaries) and export them directly to PDF or structured CSV formats.
* **Technical Dependencies:**
  - `jspdf` or `@react-pdf/renderer` for PDF generation on the client/edge.
  - `papaparse` or native JS blobs for CSV generation.
* **Architecture:** Report generation logic should reside in `src/services/reports.ts`, potentially utilizing Supabase Edge Functions if batch calculations exceed standard serverless execution limits.

## 2. Promotion Criteria Engine (`feature/v0.2-promotion-engine`)
* **Objective:** Tie attendance thresholds and unit longevity to promotion eligibility. Allow Officers to log promotion events and automatically calculate eligibility based on database metrics.
* **Technical Dependencies:**
  - Supabase RPC (PostgreSQL functions) for complex historical attendance queries.
  - Form validation additions (Zod schemas) for Promotion Logs.
* **Architecture:** Requires a new `promotion_logs` table schema mapped to `personnel_profiles` and an addition to the `src/services/` tier.

## 3. Events & Training Calendar (`feature/v0.2-event-calendar`)
* **Objective:** Build the interactive operational calendar module where Officers can schedule upcoming formations, tactical drills, or exams, including OPORD (Operation Order) summaries.
* **Technical Dependencies:**
  - UI library for calendar rendering (e.g., `react-big-calendar` or `date-fns` for custom UI rendering).
* **Architecture:** Extend the current `events` table data fetching into a dedicated `/dashboard/events` route with specialized Officer creation modals and Cadet read-only views.

## Feedback Integration Loop
As part of the v0.2.0 lifecycle, user feedback collected via the `feedback_reports` database table (established in v0.1.0) will be triaged weekly. Hotfixes will be patched directly to `main`, while feature requests will be added to the v0.3.0 backlog.
